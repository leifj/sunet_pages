Partition and HSM Capabilities and Policies

As explained elsewhere, HSM capabilities are set at the factory and control the overall "personality" or scope of the HSM. HSM policies coincide with alterable HSM capabilities and allow those capabilities to be adjusted for the entire HSM.

Similarly, when an HSM Partition is created, it also comes with a set of capabilities and policies, based on the factory base configuration and on any HSM policy settings. Partition policies can modify the effects of some partition capabilities. This can be especially useful in multi-partition HSMs where the Owners want to emphasize different characteristics from one partition to the next, for Partition Owners and Clients with different needs.

Some HSM policies are destructive, meaning that if you change the policy, the HSM must be initialized (destroying all partitions and contents) before it can be used again. Partition policies are generally not destructive. This can be illustrated by the ability to clone objects to another HSM or partition.

Example of related HSM and Partition Policies

The cloning policy is destructive at the HSM level.

The policy is not destructive at the partition level.

The HSM-level switch is “all or nothing” for the HSM and for any partitions within that HSM.

Think of it as:

  • at the HSM level the setting controls ABILITY to clone (or not)
  • at the partition level, the setting controls PERMISSION to use the ability if the ability is there.

The first one is security-critical and therefore must be destructive if changed. The second one is completely dependent on the first, and therefore does not need to be destructive.

Use Case Example

A possible use case is where a customer wants to require the HSM Security Officer (SO) to be involved with every backup operation, so the partition cloning policies are set to ‘0’ normally, and the SO must set those policies to ‘1’ before the backup can be performed.
