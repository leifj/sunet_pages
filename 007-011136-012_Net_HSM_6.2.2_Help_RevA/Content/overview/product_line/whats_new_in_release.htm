What's New in Current Release

The following are features or highlights of SafeNet HSM release 6.3.

HA Auto Reconnect

The HA feature now includes the ability of the Client to reconnect and resume application operation, without intervention, after completely losing contact with the HA group of HSMs. Context is retained, and token object states are preserved. Login is automatic.

[Requires SafeNet HSM Client 6.2.2; no firmware dependency]

HA Failover works when a member partition is deactivated

The following scenarios are addressed in 6.2.2 Client:

•When an application starts up the HA system ensures that the partition is activated.

•If a member has failed, during recovery the system checks that the partition is activated and confirms activation before the member is added back to the group.

•When a new member is being added to an HA group, system checks whether or not the new member partition is activated and subsequently ensures activation before the member is successfully added to the HA group.

Network HSM gets OpenSSL Upgrade

The SafeNet Network HSM receives OpenSSL 1.0.2i to address the vulnerability described in CVE-2016-6304.

[Requires SafeNet Network HSM appliance software version 6.2.2; no client or firmware dependency]

HA Token Objects Recovery

HA can now recover from all physical members of a group going missing, such that all token objects are recovered when members return and the group resumes working. Session objects are not recovered, and must be re-created by the application(s). This capability exceeds standard PKCS#11; therefore it is a configurable option.

Re-image Network HSM Appliance

The SafeNet Network HSM appliance can now be re-imaged back to the factory-shipped version (currently version 5.4.7)

Previous Technology Preview

The items in this section were part of the Technology Preview for release 6.2.1, and are now confirmed features of release 6.3.0.

One-step NTLS setup

Gemalto has developed a simplified, streamlined method to configure SafeNet Network HSM appliances for network trust link (NTL), using one command. See [Step 7] Create a trusted link and register Client and Appliance with each other.

Appliance-Level Logging

Appliance-side logging of HSM activity moves HSM logging directly into the appliance file system. The purpose is to record HSM operations while freeing the HSM to perform cryptographic operations. Appliance-side HSM Audit Logging and Appliance-side Audit Logging Compared with Secure Audit Logging.

Custom Roles

Named roles can be customized, allowing compartmentalization of HSM management. A custom role is defined by a simple list of the commands that it can run. The role is then applied to named users to give them only the required access. Roles can be removed from a user or added to a user. user role import   

Appliance Version via the API

To aid in provisioning and redeployment, customers managing multiple HSMs can now directly query the API to discover the version of any HSM appliance. This removes the need to open an SSH session and run shell commands, allowing more convenient management and smoother automation.

Port Bonding All Modes

HSM appliance port bonding is now supported for all Linux-standard port-bonding modes, where previously only mode 0 was supported. SafeNet Network HSM Appliance Port Bonding

Package Update without SO Login

Large enterprises providing crypto as a service now have ability to update the appliance without needing SO authentication.  

Audit Log Rotation

Audit logs can now be rotated, allowing uninterrupted HSM operation if the log space is permitted to approach full. audit log logappliance rotation
